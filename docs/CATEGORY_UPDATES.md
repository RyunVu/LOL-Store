# Category update scripts (backend)

This document describes all update/mutation operations related to `Category` in the backend codebase. It lists each method, its location, purpose, inputs, outputs, side-effects, and how to run or test it.

Location summary
- Repository interface: `backend/LoLStore.Services/Shop/ICategoryRepository.cs`
- Implementation: `backend/LoLStore.Services/Shop/CategoryRepository.cs`
- Entity: `backend/LoLStore.Core/Entities/Category.cs`

Category entity shape
---------------------
Path: `backend/LoLStore.Core/Entities/Category.cs`

Fields (main ones used by update scripts):
- `Guid Id`
- `string Name`
- `string UrlSlug` (slug generated from `Name`)
- `string Description`
- `bool ShowOnMenu`
- `bool IsDeleted`

Important: `UrlSlug` is generated by calling `category.Name.GenerateSlug()` in the repository. That extension uses the project's slugify helper.

Mutation methods (what to document)
---------------------------------
All mutation methods live in `CategoryRepository` and are declared in `ICategoryRepository`.

1) AddOrUpdateCategoryAsync
--------------------------------
- Location: `backend/LoLStore.Services/Shop/CategoryRepository.cs` (lines ~115)
- Signature:
  Task<Category> AddOrUpdateCategoryAsync(Category category, CancellationToken cancellationToken = default)
- Purpose: Create a new category or update an existing one.
- Behavior:
  - Generates `UrlSlug` from `Name` via `GenerateSlug()` before persisting.
  - Checks whether the category with `Id` exists in the DbContext:
    - If exists: sets entity state to Modified (update).
    - Else: adds a new Category entity.
  - Calls `_context.SaveChangesAsync(cancellationToken)` to persist changes.
  - Returns the saved `Category` instance (with Id assigned for new records).
- Side-effects: Writes to `Categories` table, may update the UrlSlug for the category.
- Notes & edge-cases:
  - `GenerateSlug()` must produce a unique slug for URLs; uniqueness is not enforced here — use `IsCategoryNameExistedAsync` to check duplicates before calling.
  - No explicit concurrency handling (optimistic concurrency tokens not present). Consider adding concurrency tokens if concurrent updates are expected.

2) SoftDeleteToggleCategoryAsync
------------------------------------
- Location: `CategoryRepository.cs` (lines ~131)
- Signature:
  Task<bool> SoftDeleteToggleCategoryAsync(Guid id, CancellationToken cancellationToken = default)
- Purpose: Toggle the `IsDeleted` flag for a category (soft-delete) and hide it from menu.
- Behavior:
  - Loads the category by id using `GetCategoryByIdAsync`.
  - If not found, returns `false`.
  - Sets `category.ShowOnMenu = false` and flips `category.IsDeleted = !category.IsDeleted`.
  - Marks entity as Modified and calls `SaveChangesAsync`.
  - Returns `true` on success.
- Side-effects: Updates the `IsDeleted` and `ShowOnMenu` fields in DB.
- Notes:
  - Soft-deleted categories are filtered out in `GetPagedCategoriesForUserAsync` (user queries) by `!c.IsDeleted`.

3) HardDeleteCategoryAsync
-----------------------------
- Location: `CategoryRepository.cs` (lines ~146)
- Signature:
  Task<bool> HardDeleteCategoryAsync(Guid id, CancellationToken cancellationToken = default)
- Purpose: Permanently delete the category row from the database.
- Behavior:
  - Executes a set-based delete using EF Core's `ExecuteDeleteAsync` on the set filtered by id.
  - Returns true if at least one row was deleted (> 0).
- Side-effects: Removes rows from the `Categories` table; cascading depends on DB foreign key configuration.
- Notes & precautions:
  - Hard delete is irreversible — ensure downstream referential integrity (products that reference categories).
  - Consider soft-deleting products or unlinking relations before hard deletion, or enable cascade rules intentionally.

4) ToggleShowOnMenuAsync
--------------------------
- Location: `CategoryRepository.cs` (lines ~153)
- Signature:
  Task<bool> ToggleShowOnMenuAsync(Guid id, CancellationToken cancellationToken = default)
- Purpose: Toggle the `ShowOnMenu` boolean for a category in a set-based manner.
- Behavior:
  - Performs an EF Core `ExecuteUpdateAsync` that flips `ShowOnMenu` to `!ShowOnMenu` for the matching id.
  - Returns true if at least one row was updated.
- Side-effects: Updates `ShowOnMenu` column without loading the full entity.

5) IsCategoryNameExistedAsync (helper validation)
-------------------------------------------------
- Location: `CategoryRepository.cs` (lines ~162)
- Signature:
  Task<bool> IsCategoryNameExistedAsync(string name, Guid? excludeId, CancellationToken cancellationToken = default)
- Purpose: Check whether a slugified category name already exists, optionally excluding a given Id (useful when editing to avoid false positives).
- Behavior:
  - Computes `slug = name.GenerateSlug()` and checks for any Category where `Id != excludeId` and `UrlSlug.Equals(slug)`.
  - Returns true if duplicate exists.

How these updates are called
---------------------------
- The `CategoryRepository` exposes the methods via `ICategoryRepository` which should be injected into services or endpoints.
- Currently, the `CategoryEndpoints` in `backend/LoLStore.API/Endpoints/CategoryEndpoints.cs` only exposes GET endpoints. Update/mutation endpoints (POST/PUT/DELETE) for admin/manager actions are not present in that file — they may exist elsewhere or are expected to be added.

Example usage (in code)
------------------------
Inject `ICategoryRepository` into a controller, endpoint, or service and call the method. Example (pseudo-code):

```csharp
[HttpPost("/api/admin/categories")]
public async Task<IActionResult> CreateOrUpdate([FromBody] CategoryDto dto)
{
    var category = new Category { Id = dto.Id, Name = dto.Name, Description = dto.Description, ShowOnMenu = dto.ShowOnMenu };
    var result = await _categoryRepository.AddOrUpdateCategoryAsync(category);
    return Ok(result);
}
```

- Test cases:

  - Update existing category name and assert `UrlSlug` changed accordingly.

Implementation notes & gotchas
----------------------------
- Slug generation: `GenerateSlug()` is used to create `UrlSlug` — ensure the slugify implementation is deterministic and consistent.
- Uniqueness: Repository does not enforce slug uniqueness at DB level. Consider adding a unique constraint/index on `UrlSlug` in migrations to prevent duplicates at the database layer.
- ExecuteUpdate/ExecuteDelete: These EF Core methods run set-based operations. Ensure your EF Core version and database provider support them.
- Transactions: `AddOrUpdateCategoryAsync` uses `SaveChangesAsync` — for multi-step operations across tables, consider wrapping in explicit transaction.
